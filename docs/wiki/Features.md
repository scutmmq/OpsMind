# ✨ 功能详解

## 1. 智能问答（Chat）

面向最终用户的 RAG 增强对话系统。

### 工作流

1. 用户选择知识库，创建或进入已有会话
2. 发送问题 → 后端执行 RAG 管道（详见 [🧠 RAG 引擎](RAG-Engine)）
3. SSE 流式返回：管道步骤进度事件 + token 级文本流
4. 生成完成后，用户可反馈"已解决"或"未解决"
5. 置信度低于阈值时，引导用户提交运维申告

### SSE 事件类型

| 事件类型 | 说明 |
|----------|------|
| `pipeline_step` | 管道步骤进度（如 `rewriting` → `searching` → `generating`） |
| `content` | LLM 生成的 token 级文本 |
| `done` | 生成完成，附带引用来源和置信度 |
| `error` | 管道错误（含错误码和人类可读信息） |

### 会话管理

- 会话与知识库绑定（一个会话只查一个 KB）
- 支持多轮对话，历史消息上下文传递给 LLM
- 会话列表展示，可查看历史问答记录

## 2. 知识库管理（Knowledge Base）

### 知识库层级

```
知识库 (KB)
  ├── 文章 1（手动创建 or 文档上传）
  │   ├── 分块 1 → embedding → pgvector
  │   ├── 分块 2 → embedding → pgvector
  │   └── ...
  ├── 文章 2
  └── ...
```

### 知识库 CRUD

- 创建知识库：指定名称、描述、关联的 LLM 配置
- 编辑/删除知识库（删除时清理关联向量数据）
- 每个知识库可绑定独立的 LLM 配置（不同 KB 可用不同模型）

### 文章生命周期

```
草稿(1) → 提交审核 → 待审核(2)
                        ├→ 审核通过(3) → 发布(4) → 已发布 → 停用(0)
                        │                              ↓
                        └→ 驳回(5)                  重新启用 →
                                                     重跑分块→embedding→pgvector
```

核心约束：
- 审核人不能是文章创建人（自审禁止）
- 发布时自动执行：分块 → embedding → pgvector 写入
- 停用时清理 pgvector 向量数据
- 重新启用时重新执行全部分块和向量化流程

### 文档上传

- 支持格式：**PDF、DOCX、MD、TXT**
- 上传后进入后台异步处理队列
- 处理状态可实时查询（`pending → parsing → chunking → embedding → indexing → completed`）
- 处理失败可重试

## 3. 申告管理（Ticket）

运维申告全生命周期管理。

### 状态机

```
待处理(1) ──→ 处理中(2) ──→ 已解决(4)
    │             │
    │             └──→ 需补充信息(3) ──→ 处理中(2)
    │                                        │
    └────────────────────────────────────────┘
                 │
            已关闭(5) [手动关闭或 7 天自动关闭]
```

### 关键规则

- **严格状态转换校验** — Service 层使用 CAS（Compare-And-Swap）更新：`WHERE id=? AND status=?`，防止并发操作产生脏数据
- **自动关闭** — 调度器每小时执行一次：持续 7 天未操作的申告（`status IN (1,2,3)`）自动标记为已关闭
- **补充信息限制** — 最多 3 次索要补充信息
- **处理记录** — 每次状态变更自动创建 `ticket_records` 记录

### 门户端 vs 后台端

| 操作 | 门户端（报障人） | 后台端（运维人员） |
|------|:---:|:---:|
| 提交申告 | ✅ | ✅ |
| 查看申告详情 | ✅（仅自己的） | ✅（全部） |
| 修改紧急程度/描述 | ✅（待处理时） | — |
| 开始处理 | — | ✅ |
| 索要补充信息 | — | ✅ |
| 补充信息 | ✅（被索要时） | — |
| 解决/关闭 | — | ✅ |

### 知识候选

- 运维人员在解决申告时可将处理过程一键生成为知识文章候选
- 候选文章进入知识库审核流程
- 形成"申告 → 解决 → 知识沉淀"闭环

## 4. 用户与权限（RBAC）

### 预设角色

| 角色 | 典型权限 | 适用人群 |
|------|----------|----------|
| **系统管理员** | 全部后台权限、角色/权限管理、LLM 配置、系统配置 | IT 负责人 |
| **运维人员** | 申告查看/处理、回访记录、知识候选提交 | 一线运维 |
| **知识库管理员** | 知识 CRUD、知识审核、文档上传、知识库配置 | 知识管理岗 |
| **报障人** | 智能问答、申告提交、进度查询（仅门户端） | 普通员工 |

### 权限模型

- **认证：** JWT 双令牌（access_token 2h + refresh_token 7d）
- **鉴权：** RBAC 中间件，权限码常量定义 + 角色-权限关联
- **密码策略：** 8-32 位，必须包含大写字母、小写字母和数字，bcrypt 哈希存储
- **菜单渲染：** 前端根据角色权限动态渲染侧边栏菜单

### 权限码

| 权限码 | 含义 |
|--------|------|
| `user:manage` | 用户与角色管理 |
| `ticket:read` | 申告查看 |
| `ticket:write` | 申告操作 |
| `ticket:manage` | 申告全管理 |
| `knowledge:read` | 知识库查看 |
| `knowledge:write` | 知识库编辑 |
| `knowledge:create` | 知识创建（从申告生成） |
| `knowledge:review` | 知识审核 |
| `knowledge:manage` | 知识全管理 |
| `dashboard:read` | 数据看板 |
| `audit:read` | 审计日志 |
| `system:config` | LLM 与系统配置 |

## 5. 数据看板（Dashboard）

后台首页提供运维运营数据全貌：

- **统计卡片** — 今日申告 / 待处理 / 处理中 / 已解决 / 今日问答 / 平均置信度 / 知识条目数
- **趋势图** — 按日/周粒度展示申告量和问答量变化趋势
- 数据实时刷新，无需手动更新

## 6. LLM 配置管理

在后台管理界面中动态配置，即时生效，无需重启：

- **提供商选择：** llama.cpp 本地部署 或 OpenAI-compatible API（OpenAI / DeepSeek / Moonshot 等）
- **独立配置：** LLM 和 Embedding 各自独立的 Base URL、API Key、模型名称
- **热替换：** 配置修改通过 `atomic.Value` 原子操作即时生效，不影响正在进行的请求
- **测试连接：** 验证 Base URL 可达性和模型可用性
- **默认配置：** 支持标记一个配置为默认，新创建的知识库自动使用

## 7. 审计日志

完整记录关键操作，满足合规要求：

- **记录范围：** 用户管理、知识管理、申告管理的增删改操作
- **记录内容：** 操作人、操作时间、操作类型、目标资源、变更前后数据
- **查询支持：** 按时间范围、操作人、操作类型筛选
- **不可篡改：** 审计日志只追加，不提供修改和删除接口

## 8. 站内消息

- 状态变更通知（申告状态变化时自动推送）
- 申告被索要补充信息时通知报障人
- 消息列表展示，支持已读/未读状态

## 9. 自动调度器

内置于后端进程的轻量调度器：

- **自动关闭申告：** 每小时执行，关闭持续 7 天未操作的申告
- **事务安全：** 单事务中原子执行关闭 + 创建操作记录
- **优雅关闭：** 服务停止时等待正在执行的任务完成
